# 第二版智能体本地运行隔离修复设计

日期：2026-07-15

## 问题与证据

教师端点击“生成活动方案”时返回 HTTP 500 和“服务器内部错误”。调查确认，浏览器和接口访问的并不是同一套第二版服务：

- `5173`、`8080`、`8001` 由 `第一版/integrations/buyilehu-music-agent` 中的旧进程占用。
- `5174` 由另一个 Git worktree 中的前端进程占用。
- 旧 Java 服务连接 `buyilehu_music_agent`，第二版启动配置应连接 `buyilehu_music_agent_v2`。
- 第二版 Python 尝试绑定已占用的 `8001` 时会退出，但启动脚本随后只请求健康接口；旧 Python 返回 200 后，脚本错误地把它当作新进程启动成功。

在隔离端口 `18080` 和 `18001` 上运行当前第二版源码并连接 `buyilehu_music_agent_v2` 后，同一生成接口返回 HTTP 200，成功创建互动包。因此业务生成链路本身可用，根因是本地启动过程缺少端口和进程归属隔离。

## 目标

让第二版在 macOS 本地启动时满足以下条件：

1. 启动前发现目标端口被其他进程占用时立即失败，并显示端口、PID、进程工作目录和处理建议。
2. 不再把旧服务的健康响应误判为本次启动成功。
3. 只停止由当前第二版启动脚本记录的进程，不使用宽泛的 `pkill` 误伤其他项目。
4. 正常启动后，教师端、学生端、Java、Python 均来自 `/Users/shishangbo/codex/第二版`，Java 使用 `buyilehu_music_agent_v2`。
5. 教师默认验收账号可以完成教案列表读取和互动包生成。

## 方案选择

### 采用：端口前置检查、PID 文件和启动后进程存活验证

在现有脚本结构上做小范围增强：

- 新增可复用的端口检查脚本函数，在启动任何组件前检查 `5173`、`5174`、`8001`、`8080`。
- 端口被占用时，通过 `lsof` 输出占用者 PID 和当前工作目录，随后非零退出。
- 启动 Python 和两个 Vite 前端后记录 PID 到 `.runtime/*.pid`。
- 健康检查期间同时验证刚启动的 PID 仍然存活；进程已退出时直接打印对应日志并失败，不能继续请求可能属于旧进程的健康接口。
- 新增停止脚本，只读取第二版 `.runtime` 下的 PID 文件并终止对应进程；PID 不存在或进程已结束时安全跳过。
- Java 仍以前台进程运行，由启动终端管理；启动前必须确认 `8080` 空闲。

这是推荐方案，因为它保留现有 URL 和工程结构，同时从启动边界消除串版本问题。

### 不采用：只手动停止现有进程

可以暂时恢复功能，但启动脚本的错误判断仍然存在，之后再次启动不同副本时会复发。

### 不采用：第二版永久改用其他端口

能绕开当前冲突，但会改变前端代理、README 和用户访问地址，也无法阻止未来新端口再次被其他副本占用。

## 组件与数据流

启动顺序如下：

1. 预检运行时依赖。
2. 检查四个应用端口是否空闲。
3. 启动本地 MySQL，并确认 `3306` 可访问。
4. 启动第二版 Python，记录 PID，同时检查 PID 存活和 `/api/v1/health`。
5. 启动第二版 Java，显式传递 `DB_NAME=buyilehu_music_agent_v2` 和第二版 Python 地址。
6. 启动两个第二版 Vite 前端，记录 PID并检查进程没有立即退出。
7. 教师端通过 `/api/v1` 代理访问第二版 Java；Java 通过配置地址访问第二版 Python。

端口检查只读取本机进程信息，不自动杀死未知占用者。首次修复当前环境时，可以人工确认后停止已经识别出的旧开发进程，再启动第二版。

## 错误处理

- 端口占用：打印占用端口、PID、命令和工作目录，退出码非零。
- Python 或 Vite 启动后立即退出：打印对应 `.runtime/*.log` 尾部内容并退出。
- 健康检查超时：终止本次脚本启动的 Python 进程，保留日志供诊断。
- PID 文件陈旧：若 PID 已不存在，删除陈旧 PID 文件并继续停止流程。
- PID 已被系统复用：停止前校验进程工作目录属于当前第二版，校验失败则拒绝终止并报告。

## 测试设计

先添加失败测试，再修改启动脚本：

1. 端口空闲时检查函数成功。
2. 端口被测试进程占用时检查函数失败，并包含 PID 与工作目录。
3. 被启动子进程立即退出时，健康等待不能被同端口的其他服务欺骗。
4. 停止脚本只终止 PID 文件指向且工作目录属于第二版的进程。
5. 陈旧或不属于第二版的 PID 文件不会导致误杀。

脚本测试使用临时目录、临时端口和测试进程，不依赖或终止真实开发服务。

## 验证标准

修复完成后必须取得以下新鲜证据：

1. 脚本回归测试全部通过。
2. Python 测试与 Java Maven 测试通过。
3. 两个前端构建通过。
4. `5173`、`5174`、`8001`、`8080` 的监听进程工作目录全部属于第二版。
5. Java 健康接口和 Python 健康接口均返回 200。
6. 默认教师账号登录成功，读取第二版数据库中的已解析教案。
7. 调用生成接口返回 200，响应包含新的 `packageId`、`versionId` 和 `status=success`。

## 范围边界

本次不改互动包生成业务逻辑、不引入新的模型供应商、不迁移数据库内容，也不清理第一版或其他 worktree。仅修复第二版本地启动隔离、进程管理和相关文档。
